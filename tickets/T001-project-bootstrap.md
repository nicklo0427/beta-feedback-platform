# Title

T001 - Project Bootstrap

## Goal

定義 `beta-feedback-platform` 的整體專案初始化目標、初始化順序與最低可用骨架原則，作為後續前端與後端初始化工作的共同依據。

## Background

目前專案已存在以下根目錄結構：

- `frontend/`
- `backend/`
- `tickets/`
- `README.md`
- `PRD.md`

產品定位已收斂為「跨平台 Beta 測試媒合與回饋管理平台」，不是互刷平台、不是評論交換平台、不是灌量工具。MVP 第一階段平台為 Web / H5 / PWA、iOS、Android；第二階段平台為 Steam / Desktop / Extension。

目前此 ticket 的目標不是直接進入商業功能開發，而是先建立後續可持續開發的基礎骨架與初始化共識，避免前後端各自演進導致結構失衡、命名混亂或過早複雜化。

## Scope

- 盤點目前專案根目錄既有內容與用途
- 定義專案初始化的階段目標與順序
- 明確區分「初始化工作」與「業務功能開發」
- 定義前後端初始化時應共同遵守的基本原則
- 為後續 `T002-frontend-init`、`T003-backend-init`、`T004-project-structure-rules` 提供共同上下文
- 確認初始化階段不引入與 MVP 無關的系統模組

## Out of Scope

- 不實作任何商業功能模組
- 不定義過細的 UI / UX 細節
- 不擴張到支付、聊天、通知、推薦系統等非 MVP 功能
- 不直接初始化 `frontend/` 或 `backend/`
- 不處理部署、自動化基礎設施或正式營運環境細節

## Acceptance Criteria

- 已明確列出目前根目錄已存在的資料夾與文件
- 已明確定義本專案初始化階段的主要目標是「建立可持續開發的基礎骨架」
- 已說明初始化工作應優先處理的範圍，包括前端初始化、後端初始化與結構規範
- 已明確說明初始化階段不應直接進入業務頁面與業務模組開發
- 已明確指出後續 ticket 執行時需遵守產品定位，不可偏向互刷、灌量或交換評論
- 已形成可作為後續 ticket 共同依據的初始化準則

## Deliverables

- 一份可供後續執行的專案初始化說明
- 一份初始化順序建議，至少包含前端初始化、後端初始化、結構規範三個面向
- 一組初始化階段共通約束，供後續 ticket 實作時引用

## Notes / Constraints

- 專案名稱為 `beta-feedback-platform`
- 初始化階段必須優先建立清楚、可維護、可擴充的骨架
- 初始化工作的判斷標準應偏向「是否支援後續穩定開發」，而不是「是否快速堆疊功能」
- 所有後續模組設計都應與 PRD 一致，並以真實測試與可執行回饋為方向
- 後續 ticket 若需建立檔案或初始化框架，應先遵守本 ticket 所定義的邊界，不應自行擴張需求
